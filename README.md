# 연습 문제: 신문 배달부와 지갑 (Paperboy & Wallet)

디미터 법칙 · Tell, Don't Ask · 캡슐화를 몸으로 익히는 연습입니다.
아래 요구사항과 제약을 모두 만족하도록 **직접 구현**하세요. (언어: Kotlin)

## 배경 시나리오

신문 배달부(Paperboy)가 구독 고객(Customer)에게 신문값을 받으러 다닙니다.
배달부는 고객에게 요금을 청구하고, 고객은 자신의 지갑(Wallet)에서 돈을 꺼내 지불합니다.
배달부는 하루 동안 여러 고객을 돌며 수금한 금액을 정산합니다.

> 절대 하면 안 되는 것: 배달부가 고객의 지갑을 직접 열어 잔액을 확인하고 돈을 빼가는 것.
> 현실에서 이상하듯, 코드에서도 이상합니다. 이 "이상함"을 코드로 못 하게 막는 게 목표입니다.

## 학습 목표

1. **디미터 법칙**: 한 객체가 낯선 객체(고객의 지갑)의 내부까지 파고들지 않게 한다.
2. **Tell, Don't Ask**: 상태를 물어보고 바깥에서 처리하지 말고, 객체에게 시킨다.
3. **캡슐화**: 각 객체가 자기 데이터와 규칙을 스스로 책임진다.

## 도메인 개념

| 개념 | 종류 | 책임 |
|---|---|---|
| `Money` | 값 객체(불변) | 금액을 표현. 음수 불가. 더하기/빼기는 새 `Money`를 반환 |
| `Wallet` | 엔티티 | 현금을 보관. 넣기/꺼내기를 스스로 관리 |
| `Customer` | 엔티티 | 자기 지갑을 가짐. 요청받은 금액을 지불 |
| `Paperboy` | 엔티티 | 고객에게 청구하고, 수금 총액을 정산 |

## 기능 요구사항

1. 배달부는 **한 명의 고객에게 요금을 청구**해 지불받을 수 있어야 한다.
2. 고객은 청구받은 금액을 **자기 지갑에서 꺼내 지불**한다.
3. 배달부는 **여러 고객에게서 수금**하고, 그날 **받은 총액을 조회**할 수 있어야 한다.
4. 지갑에 **돈을 충전(입금)**할 수 있어야 한다.

## 설계 제약 (이걸 지키는 게 이 연습의 핵심)

- **금지 1**: `Wallet`의 잔액을 그대로 노출하는 getter를 만들지 마라.
  (`wallet.balance` 같은 조회로 바깥에서 차감/검증하면 실패)
- **금지 2**: `Customer`가 `Wallet`을 외부에 반환하지 마라.
  (`customer.wallet` 접근 자체가 불가능해야 한다 — 지갑은 고객만 만진다)
- **금지 3**: 점(`.`)을 두 번 이상 이어 낯선 객체로 넘어가지 마라.
  (`customer.getWallet().subtract(...)` 같은 기차 충돌 금지)
- **금지 4**: 결제/잔액 부족 판단 같은 규칙을 `Paperboy`나 호출부에 두지 마라.
  규칙은 그 데이터를 가진 객체(`Wallet`/`Customer`) 안에서 강제한다.
- **불변식 1**: 잔액은 절대 0원 미만이 될 수 없다.
- **불변식 2**: `Money`는 음수가 될 수 없다(생성 시점에 거부).
- **값 객체 규칙**: `Money`는 불변. 연산은 자신을 바꾸지 않고 새 인스턴스를 반환한다.

## 공개 API 힌트 (시그니처는 스스로 다듬어도 됨)

세부 구현은 비워둡니다. "무엇을 시킬 수 있어야 하는가"만 참고하세요.

- `Money.plus(other) / minus(other)` — 결과가 음수면 예외
- `Wallet.deposit(amount)` — 입금
- `Wallet.withdraw(amount)` — 잔액 부족이면 예외, 성공 시 꺼낸 금액 반환
- `Customer.pay(amount): Money` — 자기 지갑에서 지불 (지갑은 외부에 노출 안 함)
- `Paperboy.collectFrom(customer, price)` — 청구+수금, 받은 만큼 총액에 누적
- `Paperboy.totalCollected(): Money` — 그날 수금 총액

> 힌트: `Paperboy`는 `customer.pay(price)`만 호출하면 된다. 지갑은 이름조차 몰라야 한다.

## 예외/엣지 케이스

- 잔액보다 큰 금액 청구 → 지불 실패(예외). 이때 총액은 늘지 않아야 한다.
- 0원 또는 음수 금액 청구/입금 → 거부.
- 정확히 잔액과 같은 금액 청구 → 성공, 잔액 0.

## 완료 기준 (직접 테스트로 확인)

- [ ] 정상 결제: 잔액 10,000원 고객에게 3,000원 청구 → 성공, 잔액 7,000원, 배달부 총액 3,000원.
- [ ] 잔액 부족: 잔액 1,000원 고객에게 3,000원 청구 → 예외 발생, 잔액 1,000원 유지, 배달부 총액 변화 없음.
- [ ] 다중 수금: 고객 A에게 3,000원, 고객 B에게 2,000원 수금 → 배달부 총액 5,000원.
- [ ] 경계값: 잔액과 같은 금액 청구 → 성공, 잔액 0.
- [ ] 입금: 지갑에 5,000원 충전 후 결제 가능해짐.

## 자가 점검 (구현 후 스스로 확인)

- `Paperboy` 코드 안에 `Wallet`이라는 단어가 등장하는가? → 등장하면 실패.
- 잔액을 꺼내(get) 바깥에서 `if`로 비교하는 곳이 있는가? → 있으면 Tell, Don't Ask 위반.
- 데이터만 있고 행동이 없는 클래스가 있는가? → 있으면 빈약한 모델.
- 모든 불변식이 그 데이터를 가진 객체 안에서 지켜지는가?
